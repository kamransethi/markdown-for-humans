# Task: Fix Mermaid Rendering Concurrency Error

## 1. Task Metadata
- **Task name:** Fix Mermaid Rendering Concurrency Error (`Cannot read properties of null`)
- **Slug:** fix-mermaid-rendering-concurrency
- **Status:** shipped
- **Created:** 2026-03-06
- **Shipped:** 2026-03-06

## 2. Context & Problem
**Current state:**
- When documents with multiple Mermaid charts load, only the first chart successfully renders.

**Pain points:**
- All subsequent charts display a red error message: `Diagram Error: Cannot read properties of null (reading 'firstChild')`
- Users must manually edit each chart (like adding a space) to force it to render individually.

**Why it matters:**
- Breaks document readability for any page with more than one chart.
- Looks like a critical bug to end users despite valid syntax.

## 3. Desired Outcome & Scope
**Success criteria:**
- All Mermaid charts in a document load and render successfully on initial document open, regardless of quantity.
- No `firstChild` errors are thrown.
- Orphan cleanup logic still runs correctly to prevent memory leaks in the DOM.

## 4. Work Breakdown (Completed)
- **Root Cause Analysis:** Determined that `mermaid.render` injects invisible SVGs/divs into `document.body` to measure text nodes. Concurrent renders caused the first finished chart to fire `cleanupOrphanMermaidElements()`, which wiped the `body` of all Mermaid elements. This aggressively destroyed the measurement nodes of other charts still in the process of rendering.
- **Implementation:** Added a global `Promise` queue (`mermaidRenderQueue`) in `src/webview/extensions/mermaid.ts`.
- **Behavior Change:** All charts now wait their turn to render sequentially instead of concurrently. The DOM is kept safe for measurement, and garbage collection runs safely after each individual render completes.
